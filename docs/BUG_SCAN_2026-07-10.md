## Native Layer (dllmain.cpp)

### P0.1 — `RenderVisorQuadLayers` passes `nullptr` to `nextXrReleaseSwapchainImage` on error path
**Lines:** 2105–2106
**Issue:** When `nextXrWaitSwapchainImage` fails, the error path calls
`nextXrReleaseSwapchainImage(g_visorQuad.swapchain, nullptr)`. The OpenXR spec requires the
`XrSwapchainImageReleaseInfo*` parameter to be a valid non-null pointer. The `ri` variable is
declared at line 2111, five lines too late.
**Impact:** UB on the runtime — likely crash or validation-layer error.
**Fix:** Move the `ri` declaration before the wait call and use `&ri` on the error-recovery path.

### P0.2 — D3D11 state object creation failures produce null dereference crash
**Lines:** 862, 873, 878, 880
**Issue:** `CreateRasterizerState`, `CreateBlendState`, and `CreateDepthStencilState` (lines 862,
873, 878) have their HRESULT return values completely ignored. If any fails (device removed, OOM),
the pointer stays null, yet `g_d3d11Mask.initialized = true` is set at line 880. Every subsequent
draw call (lines 1387–1418, 1489–1520, etc.) dereferences these null state objects.
**Impact:** AV crash in-game if D3D11 device is removed or memory is exhausted.
**Fix:** Check HRESULT on each Create*State call; set `g_d3d11Mask.initialized` only after all
succeed; clean up any partially-created objects on failure.

### P1.1 — `g_d3d11Mask.session` / `device` / `context` read outside any lock
**Lines:** 1761, 1803, 1862, 2595–2597, 2638, 2672
**Issue:** `g_d3d11Mask` fields are written in `XRViewLab_xrCreateSession` and read in
`XRViewLab_xrReleaseSwapchainImage`, `XRViewLab_xrEndFrame`, and
`DrawCapturedProjectionTextures` without any mutex. Multi-threaded games (e.g. UEVR) can read
stale or torn values (on 32-bit, `double` writes are non-atomic).
**Impact:** Mismatched session checks, phantom "no session" skips, possible null dereference.
**Fix:** Dedicated mutex for `g_d3d11Mask` state, or atomic-guarded reads in the draw path.

### P1.2 — Hot-reload `LoadConfig()` writes 50+ globals while hooks may be reading them
**Lines:** 2160, 2527
**Issue:** `LoadConfig()` writes ~50 global `double`, `bool`, and `enum` variables. On x86,
`double` writes can tear. If `xrLocateViews` triggers a hot-reload (line 2527) while
`xrReleaseSwapchainImage` is mid-draw reading these globals, torn reads can produce out-of-range
vertices or incorrect geometry.
**Impact:** Intermittent geometry corruption on x86 if config changes during gameplay.
**Fix:** Atomic or mutex-guarded double/bool reads in the draw path.

### P1.3 — `DrawVisorBorderToTexture` and `DrawEdgeGuardToTexture` don't check `context` before use
**Lines:** 1255, 1437, 1546
**Issue:** Entry checks verify `g_d3d11Mask.initialized` but not `g_d3d11Mask.context`. If the
context was nulled by `ReleaseD3D11MaskRenderer()` racing with a draw call, the `Map`/`Draw` calls
crash.
**Impact:** AV crash on device-loss or session-destroy timing.
**Fix:** Add `g_d3d11Mask.context` null check before draw operations.

### P1.4 — `BuildProjectedPartnerVisorVerts` uses raw scale values instead of NDC extent
**Lines:** 1197–1198, 1213–1214
**Issue:** Sets `hw = sw`, `hh = sh` (just the dimensionless `visorSize * visorWidth/Height`
values). Produces coordinates in `[-hw, hh]` × `[-hh, hh]` and feeds them to `NdcToTanX`/`NdcToTanY`
which expect `[-1, 1]` NDC. When `visorSize < 1.0`, the projected partner boundary covers a
different angular extent than the primary visor opening.
**Impact:** Partner-eye boundary misaligns with the main visor opening at small visor sizes. Left
as P1 because the partner boundary only draws in closed-bean mode.
**Fix:** Compute proper NDC extent relative to the bounding box.

### P1.5 — nullptr in SaveExperimentalSettings writes `mask_rounded` from wrong checkbox
**Lines:** `MainWindow.cs:1341`
**Issue:** `SaveExperimentalSettings` writes `mask_rounded` using `MaskEnabledCheck.IsChecked`
(the visor enable checkbox) instead of `MaskRoundedCheck.IsChecked`. The ini key `mask_rounded`
always mirrors `mask_enabled`.
**Impact:** Per-app profile fallback for `mask_rounded` (reads this ini key, line 1864) gets the
enable state instead of roundness. UI hardcodes `MaskRoundedCheck.IsChecked = true` (line 936) so
there's no correct persistence path at all — the checkbox state is never loaded from or saved to
the ini correctly.
**Fix:** Use `MaskRoundedCheck` in line 1341. Also add a `LoadSettings` read for this key.

### P1.6 — `ReadAppProfile` `mask_corner` global fallback is 0.0 while DLL default is 1.0
**Lines:** `MainWindow.cs:1834`
**Issue:** `ReadAppProfile` falls back to `0.0` for `mask_corner` when the INI key is missing.
The DLL `LoadConfig()` defaults `mask_corner` to `1.0`. Three different defaults across UI load
(0.5), per-app fallback (0.0), and DLL (1.0).
**Impact:** Preview shows maximally square corners while native renders fully rounded — a 100%
mismatch.
**Fix:** Unify to one default value (0.5 per bundled ini).

### P1.7 — Per-app profile window `_initialized = true` set before slider assignments
**Lines:** `ProfileWindow.cs:86`
**Issue:** `_initialized = true` is set at line 86, immediately after `InitializeComponent()` but
*before* slider Value assignments at lines 127–136. Every slider assignment fires `ValueChanged`
handlers (e.g. `MaskShapeSlider_Changed` at line 384) which pass the `_initialized` check and call
`SyncMaskEditorFromSliders()` with partially-initialized slider state. The preview redraws 8+
times during construction with wrong intermediate values.
**Impact:** Construction-time flicker, redundant geometry computation, potential stale state if a
handler reads another not-yet-set slider.
**Fix:** Move `_initialized = true` to after all manual init (after line 138).

### P1.8 — `UseGlobalVisor_Changed` loads global slider values but never updates preview
**Lines:** `ProfileWindow.cs:421–441`
**Issue:** When the user checks "Use global", the handler loads all global slider values (lines
428–438) but never calls `SyncMaskEditorFromSliders()`. The BeanMaskEditor preview continues to
show the pre-toggle shape. The `UseGlobal_Click` button handler (line 373) does call
`SyncMaskEditorFromSliders()`, creating an inconsistency.
**Impact:** Preview out of sync with sliders when toggling the checkbox. User could save wrong
values.
**Fix:** Add `SyncMaskEditorFromSliders()` and `UpdateHints()` after the global-value-load block.

### P1.9 — Unchecking "Use global" in profile window loses original custom values
**Lines:** `ProfileWindow.cs:421–441`
**Issue:** When the user checks "Use global" (lines 428–438 overwrite all sliders with global
values) and then unchecks it, the original per-app values loaded at construction are lost — there
are no backup fields. The sliders now display the stale global values, and any save writes those
as custom overrides.
**Impact:** Profile shape controls are silently replaced with global values even after unchecking
"Use global". User must re-enter their custom values.
**Fix:** Store per-app values in private backup fields at construction time; restore them when
unchecking.

## P2 — Medium

### P2.1 — Texture pointer held across mutex release (use-after-free risk)
**Lines:** 1549–1562, 1762–1770, 1806–1814, 1864–1872
**Issue:** The texture pointer `ts.textures[ts.lastAcquiredIndex]` is copied inside the swapchain
mutex but used (RTV creation and mapping) after the mutex is released. If `xrDestroySwapchain`
runs from another thread and frees the texture, the draw operates on a dangling pointer.
**Impact:** Rare race-condition crash if swapchain is destroyed during a concurrent frame.
**Fix:** Move all RTV/Map/Draw operations inside the mutex, or hold a ref until done.

### P2.2 — ConfigPath returns relative path when module resolution fails
**Lines:** 363–401
**Issue:** If `GetModuleHandleExA` or `GetModuleFileNameW` fails, `GetLayerDirectory()` returns
`{}`. `LegacyConfigPath()` then returns `ConfigFileName` alone — a relative path. The ini
read/write functions resolve relative paths against the game's working directory, not the DLL
directory. Config loading silently fails; all settings fall back to defaults.
**Impact:** Can't detect config changes; all settings use hardcoded fallbacks.
**Fix:** Fall back to an absolute known-good path (e.g. `%LOCALAPPDATA%\XR ViewLab\`).

### P2.3 — 49KB stack allocation in draw path
**Lines:** 1276–1277
**Issue:** `VisorVertex verts[kMaxVerts]{};` where `kMaxVerts = 4096` and `sizeof(VisorVertex) = 12
bytes` = 49KB on the stack. Called from the D3D11 draw thread. Windows default stack is 1MB, but
on constrained threads (e.g. Windows thread pool with 128KB stack default) this overflows.
**Impact:** Stack overflow on non-standard threads.
**Fix:** Use `std::vector` or reduce `kMaxVerts` to something closer to typical usage (most frames
use < 500 verts).

### P2.4 — 32-bit layer files installed to 64-bit Program Files
**Lines:** `Product.wxs:116–129`
**Issue:** 32-bit layer DLL and JSON are inside a `Win64="yes"` component under
`ProgramFiles64Folder`, placing them in `C:\Program Files\`. The `RegEnable32` component (line
168, `Win64="no"`) writes the file path to the WOW6432Node registry. When a 32-bit game's OpenXR
loader reads this path, WOW64 filesystem redirection maps it to `C:\Program Files (x86)\...`
where the files don't exist. The 32-bit layer is invisible to 32-bit games.
**Impact:** 32-bit games don't get ViewLab applied.
**Fix:** Put 32-bit files in a `Win64="no"` component under `ProgramFilesFolder`.

### P2.5 — No `$LASTEXITCODE` checks after native commands in build.ps1
**Lines:** `build.ps1:74, 103, 105, 126`
**Issue:** `$ErrorActionPreference = "Stop"` (line 7) does NOT catch native EXE failures in
PowerShell 5.1. `dotnet publish` (line 74) and the three `msbuild` invocations (lines 103, 105,
126) are never checked with `if ($LASTEXITCODE -ne 0) { throw ... }`. If any fails, the script
continues with stale or missing outputs.
**Impact:** Broken MSI built from failed steps without the script noticing.
**Fix:** Add explicit `$LASTEXITCODE` checks after each native command.

### P2.6 — `build.ps1 -Configuration Debug` produces mixed Release artifacts
**Lines:** `build.ps1:1–4, 74, 77, 90–91`
**Issue:** The script accepts a `-Configuration` parameter but publish path, native DLL paths, and
MSI source paths are hardcoded to `Release`. A Debug build packages Release DLLs.
**Impact:** Misleading local testing — user thinks they're testing Debug but gets Release bits.
**Fix:** Make paths configurable or reject non-Release configurations.

### P2.7 — Version regex in build.ps1 breaks with pre-release tags
**Lines:** `build.ps1:22, 31–32`
**Issue:** Regex `AssemblyInformationalVersion\("(\d+)\.(\d+)\.(\d+)"\)` expects exactly three
numeric components. If AssemblyInfo.cs ever uses pre-release metadata (e.g. `"4.1.108-beta.1"`),
the match fails. `$build` computation silently produces `+1` from the wrong group.
**Impact:** Nonsensical version or `ViewLab-unknown.msi` filename.
**Fix:** Relax regex for four-component and pre-release formats.

### P2.8 — `StartMenuShortcut` KeyPath uses HKCU in a per-machine MSI
**Lines:** `Product.wxs:189`
**Issue:** The Start Menu shortcut component's KeyPath is a registry value in
`HKCU\Software\cooooked\xr-viewlab\installed`. With `InstallScope="perMachine"`, the component
reference count is stored per-machine, but the KeyPath is per-user. If User A installs and User B
launches repair, the component may be incorrectly advertised or removed.
**Impact:** Start Menu shortcut may disappear for secondary users.
**Fix:** Use HKLM or a file KeyPath.

### P2.9 — `mask_corner` has three different defaults: INI=0.5, DLL=1.0, right-click-reset=0.25
**Lines:** `xr-viewlab.ini:27`, `dllmain.cpp:2191`, `MainWindow.cs:1597` (right-click path)
**Issue:** INI default is 0.5, DLL code default is 1.0, right-click reset writes 0.25 via
`1.0 - 0.75`. A user who deletes their INI gets a completely different curve shape.
**Impact:** Curve shape changes unpredictably depending on config file presence.
**Fix:** Single source of truth for defaults — update all three to match.

### P2.10 — `mask_rounded` default mismatch: INI=1, DLL default=false
**Lines:** `xr-viewlab.ini:30`, `dllmain.cpp:2190`
**Issue:** DLL code default for `mask_rounded` is `false` (visor not rounded). INI default is `1`
(visor is rounded). A config file that drops this key flips from rounded to unrounded.
**Impact:** Visor roundness changes silently after config reset/repair.
**Fix:** Align defaults.

### P2.11 — Per-app save validates stale legacy mask text fields user cannot edit
**Lines:** `ProfileWindow.cs:309–313`
**Issue:** Save validates `maskV >= 0.01` and `maskH >= 0.01` from `MaskVerticalBox`/`MaskHorizontalBox`
text fields. These are populated from per-app `mask_vertical`/`mask_horizontal` registry keys (line
112–113). If a profile has zero values from a legacy mismatch, `TryRead("0")` = `0.0` and the
`>= 0.01` check fails, blocking the user from saving any visor changes.
**Impact:** User cannot save per-app profile changes if legacy keys are zero.
**Fix:** Remove legacy validation or fall back to 0.01.

### P2.12 — `WriteAppCustomProfile` discards `visorOffsetX`/`visorOffsetY` from ProfileWindow
**Lines:** `MainWindow.cs:2214`, `ProfileWindow.cs:341–342`
**Issue:** `WriteAppCustomProfile` parameter list includes `visorSize`, visor shape keys, but NOT
`visorOffsetX`/`visorOffsetY`. ProfileWindow `Save_Click` sets `VisorOffsetXValue` and
`VisorOffsetYValue` (line 341–342), but these are discarded — only the baked bias values survive.
**Impact:** Per-app offset values set in the profile editor are lost on save.
**Fix:** Add offset values to `WriteAppCustomProfile` parameter list and write them.

### P2.13 — `CurrentGlobalMaskValues` hardcodes `rounded = true`
**Lines:** `MainWindow.cs:2083`
**Issue:** Returns `true` for `rounded` ignoring `MaskRoundedCheck?.IsChecked`. When
`ApplyGlobalMaskValuesToProfile` resets a per-app profile, `appProfile.MaskRounded` is set to
`true` even if the global roundness checkbox is unchecked.
**Impact:** Profile reset can set `mask_rounded = true` when global state is false.
**Fix:** Use `MaskRoundedCheck?.IsChecked == true`.

### P2.14 — Nose bridge Bézier control points in BeanMaskEditor lack `dir` factor
**Lines:** `BeanMaskEditor.cs:319–322`
**Issue:** `AddNoseBridgeCurve` always computes handle offsets as positive values from `startX`
toward `endX`. For the right eye (moving from center leftward, i.e. `startX > endX`), `dx < 0`
but handles still point rightward, away from the inner edge. The native `BuildNoseBridgeCurve`
(dllmain.cpp:1020) correctly uses `dir = dx > 0 ? 1 : -1`. Currently dormant because
`BuildOpenInnerGeometry` only calls `AddOpenInnerHalf(ctx, area, outerLeft: true)`.
**Impact:** If right-eye preview is ever added, the bridge curve will be geometrically wrong
(mirrored).
**Fix:** Add `dir` factor to handle offset computation, matching the native implementation.

### P2.15 — ProfileWindow XAML assigns render content to an 18px fixed row, leaving the Auto row empty
**Lines:** `ProfileWindow.xaml:114–115, 142–144`
**Issue:** Row definition index 10 is `Height="18"` (intended spacer), row 11 is `Height="Auto"`
(intended content). But the "Render %" TextBlock + RenderBox are placed in `Grid.Row="10"` — the
18px spacer. The content is clipped vertically while the Auto row (row 11) sits empty.
**Impact:** Render scale display in profile popup may be partially invisible.
**Fix:** Swap row assignments: put content in row 11, spacer in row 10.

### P2.16 — `AppProfile.Summary` reports per-app `MaskEnabled` but `mask_enabled` is never stored per-app
**Lines:** `AppProfile.cs:345`, `MainWindow.cs:2226`
**Issue:** `AppProfile.Summary` includes `$";M {maskVertical}/{maskHorizontal}"` only when
`MaskEnabled` is true. But `WriteAppCustomProfile` always deletes `mask_enabled` from the registry
(line 2226), so the Toggle visor enable checkbox in the profile row shows state that can't actually
be overridden per-app.
**Impact:** UI shows per-app visor enable that silently falls back to global on next load.
**Fix:** Remove `MaskEnabled` from the per-app summary, or keep the stale-UI promise.

### P2.17 — ProfileWindow stencil checkbox is a disconnected local copy
**Lines:** `ProfileWindow.cs:233`, `ProfileWindow.xaml:182`
**Issue:** `MaskBeanEditor.OpenInnerPreview = StencilOuterEdgesCheck?.IsChecked == true` reads the
profile window's own `StencilOuterEdgesCheck`, which defaults unchecked. This is never initialized
from the main window's global stencil setting.
**Impact:** Profile preview always shows closed-bean shape regardless of global stencil setting.
**Fix:** Pass global stencil state to ProfileWindow constructor and initialize the local checkbox.

## P3 — Low

### P3.1 — `ApplyVisorMask` first segment is a degenerate triangle
**Lines:** 1042–1044
**Issue:** First iteration emits vertices `(prevX, yBase), (prevX, prevY), (x, y)` where
`prevX = startX` and `prevY = yBase = y0`. The first two vertices are identical, producing a
zero-area triangle. D3D11 rasterizer discards it harmlessly.
**Impact:** One wasted vertex per eye per frame.
**Fix:** Start the loop at `i = 1` or skip the first degenerate segment.

### P3.2 — `build.ps1` version regex may produce 4-component version filename
**Lines:** `build.ps1:133–135`
**Issue:** MSI filename uses version from AssemblyInfo.cs. If AssemblyInfo.cs stores
`"4.1.108.0"` (4 components), the filename becomes `ViewLab-4.1.108.0.msi` but Product.wxs stores
only 3 components. On match failure, filename becomes `ViewLab-unknown.msi`.
**Impact:** Cosmetic version mismatch between filename and MSI metadata.
**Fix:** Normalize to 3-component version.

### P3.3 — Dead code: `_ = 1;` in `CheckUpdates_Click`
**Lines:** `MainWindow.cs:283`
**Issue:** `_ = 1;` is a discard assignment that does nothing — leftover debugging artifact.
**Impact:** None (dead code).
**Fix:** Remove.

### P3.4 — Dead ternary in `DiscoverKnownSteamApps` stores non-existent paths
**Lines:** `MainWindow.cs:1703`
**Issue:** `modulePath = File.Exists(guessed) ? guessed : guessed;` — both branches return
`guessed`. When the Steam path exists but the exe doesn't, the registry gets a non-existent path.
**Impact:** Registry contains invalid module paths for some Steam apps.
**Fix:** Ternary should be `? guessed : ""`.

### P3.5 — `ReadAppProfile` reads global INI fallbacks inside per-app loop (redundant I/O)
**Lines:** `MainWindow.cs:1826–1841`
**Issue:** Every call to `ReadAppProfile` (once per app entry) reads global INI values for
fallbacks from disk. These don't change between app entries.
**Impact:** O(n) INI reads where n = app count (typically 5–15, but could be 100+).
**Fix:** Cache fallback values once before the loop.

### P3.6 — PreserveConfig.vbs last line omits trailing newline
**Lines:** `PreserveConfig.vbs:136`
**Issue:** `outFile.Write kept(i)` for the last line writes without CRLF. Some INI parsers expect
a final newline.
**Impact:** Minimal — `WritePrivateProfileString` handles this fine.
**Fix:** Use `WriteLine` for all lines.

### P3.7 — Legacy INI migration path hardcodes `%ProgramFiles%` install path
**Lines:** `PreserveConfig.vbs:21`
**Issue:** One-time legacy migration only checks `%ProgramFiles%\xr-viewlab\`. Custom install
directories are skipped silently.
**Impact:** Legacy config not migrated for custom installs.
**Fix:** Document; low priority.

### P3.8 — Eight bias/curve keys read by DLL absent from default INI
**Lines:** `xr-viewlab.ini` (missing); `dllmain.cpp:2192–2198`
**Issue:** DLL reads `mask_offset_x/y`, `mask_top/bottom/left/right_bias`,
`mask_top/bottom_curve`. None appear in the bundled INI.
**Impact:** Users can't discover these keys from documentation alone.
**Fix:** Add to INI, or mark as intentionally hidden.

### P3.9 — No contract test verifies `mask_inner_bridge_width` in per-app delete list
**Lines:** `Tests\Verify-ViewLabContracts.ps1:47–51`
**Issue:** The contract checks four visor shape keys in the per-app cleanup assertion but omits
`mask_inner_bridge_width`. (The actual PreserveConfig.vbs IS correct — the key is deleted at line
159 — but no test pins this.)
**Impact:** Future refactors that drop this delete would not be caught.
**Fix:** Add assertion for `mask_inner_bridge_width` in the per-app cleanup list.

### P3.10 — No contract test verifies DLL reading of `crop_outer_edges_only`
**Lines:** `Tests\Verify-ViewLabContracts.ps1:79–95`
**Issue:** The DLL reads `crop_outer_edges_only` at line 2165 and uses it. No contract test
asserts this read (STATE.md §5 is stale — it claims the DLL ignores it, which is wrong).
**Impact:** Refactor that removes the read would not be caught.
**Fix:** Add contract assertion (it's already implicitly in the log format check at line 102, but
an explicit key-read assertion would be safer).
